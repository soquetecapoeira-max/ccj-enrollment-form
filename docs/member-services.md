# メンバー向け手続きの拡張設計（入会フォームの延長）

入会申し込みで確立した **静的 HTML + Google Apps Script（ウェブアプリ）+ スプレッドシート** のまま伸ばすのが、運用・コスト・保守のバランスがよいです。  
このドキュメントでは **アイデア**、**データの持ち方**、**フロントと GAS のきれいな分け方**、**実装の段階** をまとめます。

**実装状況:** フェーズ1（プラン変更）は **完了**しています（`plan-change.html`、GAS の `formType: plan_change`、シート「プラン変更」への追記と申請者向け確認メール）。

---

## 1. ゴール像（自動化の範囲）

| 領域 | フォームでよいこと | 人が必ず介在した方がよいこと |
|------|---------------------|------------------------------|
| プラン変更 | 希望内容・希望開始タイミングを記録し、担当に通知 | 会費・契約の確定、PayPal／振替の変更手続き |
| 休会 | 期間・理由・連絡先を記録し、担当に通知 | 実在会員かの照合、規程に沿った承認 |
| 大会・イベント | 参加意思・付帯情報を集計し、締切前に一覧化 | 当日運営、支払い確認、キャンセル対応 |

**自動化の主役**は「入力の正規化・ログ化・通知」です。**承認と会計まですべて自動**にすると、なりすまし・誤入力・規程外のリスクが跳ね上がります。  
現実的には **「申請は即スプレッドシートへ。担当が台帳と突き合わせて処理」** が安全でスマートです。

---

## 2. 既存フォームを活かす共通パターン

現在の入会フローは次の形です。

1. ブラウザで入力 → 確認画面 → `fetch(POST)` で JSON 送信  
2. GAS `doPost` が JSON をパースし、**1行** `appendRow`  
3. 任意で `MailApp` により申込者へ確認メール  

拡張時も **この3点は変えない**。変えるのは **JSON の意味** と **書き込み先シート** だけにすると構造がきれいです。

### 2.1 推奨ペイロード規約（将来の GAS で分岐するため）

すべてのフォームで、POST ボディを次の形に寄せるとルーティングが単純になります。

```json
{
  "formVersion": 1,
  "formType": "enrollment",
  "submittedAt": "2026-04-16T12:00:00+09:00",
  "payload": {
    "氏名": "…",
    "メールアドレス": "…"
  }
}
```

| フィールド | 役割 |
|------------|------|
| `formVersion` | スキーマ変更時に GAS 側で互換分岐できるようにする |
| `formType` | `enrollment` / `plan_change` / `leave` / `event` など固定文字列 |
| `submittedAt` | クライアント時刻（参考）。行のタイムスタンプは GAS で上書きしてもよい |
| `payload` | フォーム種別ごとのキー・値（現在の入会はフラット JSON なので、移行時は GAS が両対応でもよい） |

**現状の入会フォーム**は `payload` なしのフラット JSON のままで動いています。  
拡張するタイミングで GAS を「フラット **または** `formType` + `payload`」の両受けにすると、既存ページを壊さずに移行できます。

### 2.2 スプレッドシートの持ち方

**おすすめ**: 1 ブックの **シートをタブで分ける**。

| シート名（例） | 内容 |
|----------------|------|
| `入会` | 現行どおり（ヘッダー行は現状維持） |
| `プラン変更` | 申請日時、氏名、メール、現コース、希望コース、希望開始、備考、… |
| `休会` | 申請日時、氏名、メール、休会開始希望、再開希望、理由、… |
| `大会_イベント名` | 大会ごとにシートを分けるか、1シートに「イベントID」列を設ける |

「大会ごとにシート」は見やすいがシート数が増える。「1シート + イベントID」は集計しやすい。どちらも運用で選べます。

### 2.3 GAS の構造（イメージ）

1 デプロイURLのまま、`doPost` 内で:

1. JSON パース  
2. `formType` があればそれで分岐、なければ従来どおり `入会` シートへ（後方互換）  
3. 該当シートに `appendRow`  
4. （任意）担当者メール・Slack Webhook・確認メールを `formType` ごとにテンプレ分岐  

`doGet` は運用確認用の短文のままでよいです。

## 3. フロントエンド（HTML）のきれいな置き方

GitHub Pages の URL を変えたくない場合、**ルートの `index.html` は入会のまま**にしておくのが安全です。

推奨ディレクトリ案（段階的に追加）:

```
ccj-enrollment-form/
├── index.html                 # 入会（現行・公開URLのエントリ）
├── plan-change.html           # プラン変更（会員向け・実装済み）
├── leave-of-absence.html      # 休会（追加時）
├── event-*.html               # 大会別、または event.html + クエリで大会ID
├── assets/                    # 共通CSS・小さな共通JS（任意・後から抽出）
│   ├── theme.css
│   └── form-submit.js
├── google-apps-script.js      # リポジトリ上の参照用（実体はスプレッドシート bound の GAS）
└── docs/
    └── member-services.md     # 本ドキュメント
```

- **デザインの統一**: 入会 `index.html` の `:root` とカードUIを `assets/theme.css` に切り出すと、各手続きページが同じ見た目になります（リファクタは任意）。  
- **SCRIPT_URL**: 原則 **同一の GAS デプロイ URL** を各ページで共有（ルーター1本化）。別デプロイにすると URL の更新が増えて煩雑になります。

## 4. ワークフロー別アイデア（フィールド例）

### 4.1 プラン変更（実装済み）

- **フォーム**（`plan-change.html`）: 氏名・メール・電話、現コース／希望コース（同一不可）、希望反映タイミング（翌月／翌々月／相談）、備考  
- **POST**: `formVersion`, `formType: "plan_change"`, `payload`（上記キーは日本語プロパティ名）  
- **GAS**: シート「プラン変更」に 1 行追記（シートが無ければヘッダー付きで自動作成）。申請者へ確認メール（`sendPlanChangeConfirmationEmail_`、不要なら GAS から削除可）  
- **今後足せるもの**: 担当者への別通知メール、生年月日など追加照合項目

### 4.2 休会

- 休会希望開始日、希望再開時期、理由（選択式 + 自由記述）、緊急連絡先  
- 規程で必要なら「規程を読み同意」チェック  
- **自動化**: 同上 + カレンダー用に「再開予定日」列を後からフィルタ可能に

### 4.3 大会・イベント手続き

- イベント識別子（固定文字列でよい）、参加区分、同伴者人数、食事アレルギー、Tシャツサイズ等  
- **自動化**: 締切日を別列で持ち、GAS の時間トリガでリマインダメールは可能だが設定が増える。**まずはシート上のフィルタと手動リマインド**から始めるのが現実的  
- 参加者が多くなったら **Googleフォーム** へのリンクだけ貼る選択肢もあり（集計はフォーム側、本会員の細かい手続きは HTML 側）

## 5. セキュリティ・運用メモ（短く）

- ウェブアプリは「誰でも POST できる」前提なので、**秘密情報や決済の確定操作は載せない**。  
- **会員照合**はスプレッドシート側の担当作業に寄せるか、将来のみ Google ログイン等を検討。  
- スパム対策: 簡易ハニーポット、送信レート制限（GAS Properties で粗く制御）などは余裕が出たら。
